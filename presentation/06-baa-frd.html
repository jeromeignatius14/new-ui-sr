Introduction & Background

The Rolling Block Management System (RBMS), developed by ADRIG AI Technologies Private Limited, digitises the complete block lifecycle from request creation through multi-department disconnection approvals, optimisation, and sanctioning by the Traffic Controller. RBMS is live across Southern Railway and handles 5,000+ users daily across 6 divisions.

The post-sanctioning stage — where a sanctioned block is availed at site — historically had no digital record. Field staff applied verbally to Station Masters, availing status was invisible to management, and block bursts or non-availings went untracked. A separate Block Authorization App (BAA) was required to close this gap.

An earlier version of the BAA was developed by Plattr Tech Studio (reference document: Functional Overview – RBMS Availing Part). ADRIG AI Technologies is now developing an enhanced BAA — tightly integrated with RBMS — with additional capabilities including real-time compliance tracking, structured drop-reason capture, and an analytics dashboard. This document specifies the functional requirements for that system.

Scope of This Document
In Scope (BAA): All functionality from the point a block is sanctioned in RBMS to the point the block is digitally closed and acknowledged — including SM approval, availing start/stop, time extension, closure, compliance tracking, and analytics.
Out of Scope: Block request creation, disconnection approvals, Department Controller gating, optimisation, sanctioning — all of which remain within RBMS. This document does not re-specify RBMS features.

Functional Requirements
6.1   Authentication & Access Control
IDRequirementPriority
F-01Users authenticate via CUG mobile number + OTP (same credential as RBMS). No separate login for BAA.MUST
F-02Role-based dashboard displayed on login. Each user sees only screens relevant to their designation.MUST
F-03Access is restricted to sanctioned blocks within the user's jurisdiction / section mapping.MUST
6.2   Field Staff — Avail Block Dashboard
IDRequirementPriority
F-04Dashboard displays all sanctioned blocks assigned to the logged-in user, their supervisor, and team members under the same hierarchy.MUST
F-05Each block card shows: Section, Granted Time From/To, Block Type (Corridor / Urgent / Outside Corridor), current status, and pending action indicator.MUST
F-06Audio beep alert and red visual urgency indicator triggered in-app when a sanctioned block requires action.MUST
F-07Browser-based push notification delivered to field staff when a block is sanctioned in RBMS. Notification works even when the browser tab is not in focus, using the Web Push API.MUST
F-08Personal compliance health score displayed prominently: e.g. "14 of 18 sanctioned blocks acted upon (78%)". Score updates on each action.NEW
F-09Blocks expiring without action (24 hrs past block time) trigger a mandatory "Acknowledge Expired Block" modal on next login. User must select a drop-down reason before dismissing.NEW
6.3   Availing Submission Flow
IDRequirementPriority
F-10SSE submits availing application with: work location from/to, additional remarks, and any concurrence details required from other departments.MUST
F-11Single-department blocks: Application submitted directly to SM.MUST
F-12Combined-department blocks: (a) Without SSE edits — application routes directly to SM; the dependent department field staff is notified that their sanctioned block is being availed. (b) With SSE edits to the application — the dependent department field staff must accept the changes first, then the application routes to SM. (c) If SM modifies the approved timing — the dependent department must again accept the revised timing before availing proceeds.MUST
F-13Power Blocks: Application submitted in parallel to both SM and Traction Power Controller (TPC). TPC approves first; SM grants subsequently.MUST
F-14Pre-submission time modification allowed only for reduction: blocks < 2 hours → max 30% reduction; blocks ≥ 2 hours → max 10% reduction.REF-01
F-14aGeo-fencing — Avail Submission: Device GPS location captured and validated at the time of availing submission. Submission is rejected if the device is outside the permitted radius around the declared work location.REF-01
6.4   SM Dashboard & Actions
IDRequirementPriority
F-15SM dashboard shows all pending avail requests from field staff under their station jurisdiction, with full block details visible.MUST
F-16SM can: Approve as-is; Modify start/end time and approve; Reject with mandatory reason. All actions are timestamped and attributed to SM identity.MUST
F-17On SM action, field staff receives an instant browser notification. On rejection, reason is shown prominently in the field staff's dashboard.MUST
F-18Active Blocks screen: SM sees all currently availing blocks with live elapsed time and remaining time per block. Block burst (time exceeded) shown as a red flag automatically.NEW
F-19SM receives closure acknowledgement prompt when SSE submits block return. SM must acknowledge before block is marked Closed.MUST
6.5   Field Staff Acknowledgement & Availing Start
IDRequirementPriority
F-20After SM approval, field staff must explicitly acknowledge before availing starts. If SM modified timing, SSE sees the modified time and must accept or reject the modification.MUST
F-21Only after acknowledgement can SSE tap "Start Availing". A live countdown timer is displayed from the moment availing starts.MUST
F-22Once availing is started, block status updates to "Availing In Progress" — visible to SM, Dept. Controller, and Traffic Controller in real time.NEW
F-22aGeo-fencing — Availing Start: Device GPS location captured and validated when SSE taps "Start Availing". The action is blocked if the device is outside the permitted radius around the declared work location.REF-01
6.6   Time Extension
IDRequirementPriority
F-23Time extension can only be requested after availing has started (not before). SSE must enter additional time required and provide reason.MUST
F-24Extension request routed to SM (and TPC for power blocks). SM/TPC can approve or reject with mandatory remarks.MUST
F-25If block time is exceeded without an approved extension, system auto-flags "Block Burst" on SM's active blocks screen.NEW
F-25aGeo-fencing — Time Extension Request: Device GPS location is validated when SSE submits a time extension request. The request is rejected if the device is outside the permitted radius of the work location.REF-01
6.7   Block Closure
IDRequirementPriority
F-26SSE submits closure with: completion remarks and optional image attachment. System records exact closure submission timestamp.MUST
F-27SM reviews and approves closure. For Power Blocks: TPC closes first, then SM closes other department blocks.MUST
F-28Only after SM (and TPC where applicable) approval does block status change to CLOSED. This is the terminal state — no further actions possible.MUST
F-29Closed block data (availed time, actual duration, section, km, department) transmitted to e-TSR server via API. Transmission status visible on SM Dashboard.REF-01
F-29aGeo-fencing — Closure Submission: Device GPS location captured and validated when SSE submits block closure. Closure is rejected if the device is outside the permitted radius of the work location at the time of submission.REF-01
6.8   Drop-Reason Capture  NEW — Not in Prior BAA
IDRequirementPriority
F-30When an SSE does not submit an availing application for a sanctioned block (block expires), a mandatory reason dropdown is shown. Options include: "Work already completed", "Material not ready", "Manpower unavailable", "Section not available", "SM not reachable", "Time not suitable", "Other (enter remarks)".NEW
F-31When SM rejects an availing request, SM must select a structured reason from a dropdown (in addition to free-text remarks). Options include: "Line not clear", "Train movement conflict", "Other department pending", "Safety concern", "Other".NEW
F-32All drop-reasons are stored against the block record and surfaced in the Analytics Dashboard for section-wise and stage-wise analysis.NEW
6.9   Analytics & Compliance Dashboard  NEW — Not in Prior BAA
IDRequirementPriority
F-33Section-wise funnel dashboard showing count and % at each stage: Sanctioned → Applied → SM Approved → Availing Started → Closed. Accessible to Traffic Controller and DRM.NEW
F-34Stage-wise drop-reason breakdown: for each stage transition where a drop occurs, the top reasons (from F-30, F-31) shown as a ranked list with count and %.NEW
F-35SSE-level compliance health score visible to Dept. Controller: shows each SSE's acted / total sanctioned ratio and trend over last 4 weeks.NEW
F-36All analytics filterable by: date range, section, department, block type. XLSX export available for all views.NEW

Non-Functional Requirements
IDCategoryRequirement
NF-01PlatformBAA works on any mobile browser (iOS Safari, Android Chrome) without App Store installation (PWA). Also usable on desktop.
NF-02AuthenticationIntegrated with RBMS CUG + OTP authentication. Sessions effectively persistent for field staff (no repeated logins).
NF-03NotificationsBrowser-based push notifications via Web Push API. Delivered even when the browser tab is in background. Works on both iOS Safari and Android Chrome without any native app installation.
NF-04Audit TrailEvery action (submit, approve, reject, modify, extend, close) stored with user identity, timestamp, and structured reason. Immutable audit log.
NF-05PerformanceAll dashboard screens load within 3 seconds on a standard 4G connection. Live timers update every second without page refresh.
NF-06SecurityJWT-based access token with auto-refresh. All API calls over HTTPS. Jurisdiction-based data filtering enforced at backend, not just frontend.
NF-07ScalabilitySystem must handle concurrent availing actions across all 6 SR divisions simultaneously without degradation.
